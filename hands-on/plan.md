# 收藏管理与图书列表检索增强交付计划

## 功能概述

本次交付聚焦四项现有能力补全与可用性提升：

1. 支持用户从收藏列表中移除单本书籍。
2. 支持用户一键清空全部收藏。
3. 为图书列表增加排序能力，允许按书名或作者排序。
4. 为图书列表增加实时搜索能力，允许按书名或作者筛选。

本计划偏工程交付，不扩展为推荐、分页、服务端检索等新范围。整体原则是优先复用现有 React、Redux Toolkit、Express、JSON 文件存储与现有测试栈，在不新增依赖的前提下完成闭环。

## 现状基线

- 收藏接口当前仅支持读取和新增，位于 [backend/routes/favorites.ts](../backend/routes/favorites.ts)。
- 收藏页当前只负责展示列表，没有移除单本和清空全部操作，位于 [frontend/src/components/Favorites.tsx](../frontend/src/components/Favorites.tsx)。
- 图书页当前直接渲染 books.items，没有搜索输入和排序控件，位于 [frontend/src/components/BookList.tsx](../frontend/src/components/BookList.tsx)。
- 收藏状态管理当前只有 fetchFavorites 和 addFavorite，位于 [frontend/src/store/favoritesSlice.ts](../frontend/src/store/favoritesSlice.ts)。
- 图书列表后端当前只返回全量图书，不支持任何查询参数，位于 [backend/routes/books.ts](../backend/routes/books.ts)。
- 当前自动化测试覆盖新增收藏与基础浏览路径，但未覆盖移除、清空、搜索、排序，见 [backend/tests/favorites.test.ts](../backend/tests/favorites.test.ts) 和 [frontend/tests/e2e/book-favorites.spec.ts](../frontend/tests/e2e/book-favorites.spec.ts)。
- 当前前端没有现成的单元测试脚本或测试依赖，见 [frontend/package.json](../frontend/package.json)。

## 用户故事

- 作为已登录用户，我希望在收藏页移除某一本书，这样我可以维护我的收藏列表。
- 作为已登录用户，我希望在收藏页一键清空全部收藏，这样我不需要逐本删除。
- 作为已登录用户，我希望按书名或作者排序图书列表，这样我可以更快浏览内容。
- 作为已登录用户，我希望通过书名或作者实时搜索图书，这样我可以更快定位目标书籍。

## 交付约束与实施原则

- 不新增依赖，除非后续确认前端单测基础设施必须补齐并获得批准。
- 依赖方向遵循 types < Service < Repo < UI < Runtime。
- 以最小改动为原则，优先在现有 slice、现有路由和现有页面上增量扩展，不引入新的全局状态层级或复杂缓存策略。
- 收藏能力以后端 users.json 持久化结果为准，不能只做前端本地删除，否则刷新后会回弹。
- 搜索与排序优先纯前端实现，因为当前图书列表已由 [backend/routes/books.ts](../backend/routes/books.ts) 一次性返回全量数据，且项目使用 JSON 文件存储，不适合为小数据量场景提前扩展服务端查询契约。
- UI 变更应尽量保持当前页面结构不变，只新增必要控件和状态提示。

## 影响面拆分

| 层面 | 是否改动 | 主要内容 | 说明 |
| --- | --- | --- | --- |
| 类型 | 小幅改动 | 前端可补充排序字段、搜索输入、收藏操作状态等轻量类型；后端 Book 和 User 主结构可保持不变 | 不建议改动 Book、User 数据结构，避免迁移 users.json |
| 状态管理 | 必改 | 在 [frontend/src/store/favoritesSlice.ts](../frontend/src/store/favoritesSlice.ts) 增加 removeFavorite、clearFavorites；图书搜索和排序优先放在页面局部状态或轻量 selector 中 | 搜索和排序属于视图派生状态，不建议写回 books slice |
| 前端 UI | 必改 | 在 [frontend/src/components/Favorites.tsx](../frontend/src/components/Favorites.tsx) 增加移除按钮、清空按钮、处理中状态；在 [frontend/src/components/BookList.tsx](../frontend/src/components/BookList.tsx) 增加搜索框和排序控件 | 不需要新增页面 |
| 后端接口 | 部分必改 | favorites 需要新增删除接口；books 接口建议不改 | 收藏删除需要持久化，必须走后端；搜索排序可纯前端完成 |
| 运行时 | 小幅改动 | 路由挂载结构无需变化，继续通过现有 API router 暴露能力 | 不涉及部署拓扑变化 |

## 接口与数据流建议

### 是否需要新增后端接口

结论如下：

- 单本移除收藏：需要新增后端接口。
- 清空全部收藏：需要新增后端接口。
- 图书排序：不需要新增后端接口。
- 图书搜索：不需要新增后端接口。

原因如下：

- 收藏数据持久化在 users.json，只有前端本地删除会导致刷新或重新登录后状态恢复，无法满足真实业务闭环。
- 当前图书接口已返回全量图书，且项目没有分页、排序、搜索参数体系；在当前 JSON 文件存储架构下，把排序和搜索放到后端只会增加契约和测试成本，收益有限。

### 建议的接口形态

| 接口 | 用途 | 建议返回 | 说明 |
| --- | --- | --- | --- |
| DELETE /favorites/:bookId | 删除单本收藏 | 200 + message，或 204 无响应体 | 为了和现有 POST /favorites 风格一致，优先建议 200 + message |
| DELETE /favorites | 清空当前用户全部收藏 | 200 + message，或 204 无响应体 | 语义清晰，前端实现最直接 |
| GET /favorites | 保持不变 | 返回收藏书籍数组 | 继续作为收藏页真值来源 |
| GET /books | 保持不变 | 返回图书数组 | 不引入 sort、query 参数 |

### 建议的数据流

- 收藏单本移除与清空：
  - Favorites 页面触发删除操作。
  - favoritesSlice 发起异步请求。
  - 请求成功后，优先沿用当前模式再次调用 fetchFavorites，确保 UI 与后端持久化结果同步。
  - 本轮不建议做 optimistic update，避免在 JSON 文件写失败时引入回滚复杂度。
- 图书搜索与排序：
  - BookList 保持 books.items 作为原始数据源。
  - 搜索词和排序选项放在页面局部状态。
  - 页面根据原始图书列表派生 visibleBooks 后渲染。
  - 收藏状态仍继续依赖 favorites.items 计算，不改变现有已收藏标识逻辑。

## 里程碑

## Milestone 1：补齐单本移除收藏链路

### 目标

让用户可以在收藏页移除单本书籍，并在刷新后保持结果一致。

### 涉及改动

- 后端：
  - 在 [backend/routes/favorites.ts](../backend/routes/favorites.ts) 增加 DELETE /favorites/:bookId。
  - 保持鉴权、用户不存在、缺失 bookId、非法 bookId 的错误处理风格与现有路由一致。
  - 在 [backend/tests/favorites.test.ts](../backend/tests/favorites.test.ts) 增加删除成功、未登录、用户不存在、删除不存在收藏项等测试。
- 状态管理：
  - 在 [frontend/src/store/favoritesSlice.ts](../frontend/src/store/favoritesSlice.ts) 增加 removeFavorite 异步 action。
  - 为删除过程补充最小可用的请求状态处理，避免重复点击。
- 前端：
  - 在 [frontend/src/components/Favorites.tsx](../frontend/src/components/Favorites.tsx) 为每本书增加“移除”按钮。
  - 删除成功后刷新收藏列表；删除最后一本后切换到空状态展示。
- 类型：
  - 如有必要，在 [frontend/src/types.ts](../frontend/src/types.ts) 增加收藏操作状态类型；后端 [backend/types.ts](../backend/types.ts) 不建议改动主数据结构。

### Done when

- 收藏页每本书都有可点击的移除操作。
- 删除后当前列表立即更新，刷新页面后状态不回弹。
- 删除不存在的收藏项不会导致前端崩溃，接口行为可预测。
- 现有新增收藏能力不回归。

## Milestone 2：补齐清空全部收藏能力与页面状态完整性

### 目标

让用户能够一次性清空收藏列表，并保证空状态、重复点击和失败反馈处理完整。

### 涉及改动

- 后端：
  - 在 [backend/routes/favorites.ts](../backend/routes/favorites.ts) 增加 DELETE /favorites。
  - 统一“已为空时再次清空”的处理语义，建议保持幂等。
  - 在 [backend/tests/favorites.test.ts](../backend/tests/favorites.test.ts) 增加清空成功、未登录、用户不存在、空列表重复清空等测试。
- 状态管理：
  - 在 [frontend/src/store/favoritesSlice.ts](../frontend/src/store/favoritesSlice.ts) 增加 clearFavorites 异步 action。
  - 删除中和清空中应有最小区分能力，至少避免同时重复触发多个写请求。
- 前端：
  - 在 [frontend/src/components/Favorites.tsx](../frontend/src/components/Favorites.tsx) 增加“清空全部收藏”按钮。
  - 收藏为空时隐藏或禁用清空按钮，避免无意义操作。
  - 清空成功后直接进入现有空状态文案。
  - 若请求失败，需要至少有统一失败提示或保留现有 failed 状态出口。
- 交互：
  - 本轮不强制新增二次确认弹窗，优先保证最小闭环；如需确认，可先用浏览器原生 confirm，不引入新依赖。

### Done when

- 收藏页存在清空入口，且仅对已登录用户生效。
- 清空后列表为空，刷新页面后仍为空。
- 收藏本来为空时，页面行为稳定，没有报错或脏状态。
- 单本删除与清空操作不会互相打架，页面不会出现明显闪烁或重复写入。

## Milestone 3：为图书列表增加搜索与排序

### 目标

在不改动图书后端接口的前提下，为图书列表提供可立即使用的搜索和排序能力。

### 涉及改动

- 前端：
  - 在 [frontend/src/components/BookList.tsx](../frontend/src/components/BookList.tsx) 增加搜索输入框。
  - 增加排序选项，默认保留原始顺序，支持按书名排序、按作者排序。
  - 搜索对象覆盖书名和作者，输入时实时生效。
  - 搜索和排序共同作用于同一份 visibleBooks 派生结果。
  - 无匹配结果时增加明确空状态文案，而不是展示空白页。
- 状态管理或类型：
  - 优先使用页面局部状态管理搜索词和排序项，不把视图参数写入 [frontend/src/store/booksSlice.ts](../frontend/src/store/booksSlice.ts)。
  - 如需抽出可复用类型，可在 [frontend/src/types.ts](../frontend/src/types.ts) 增加轻量枚举或联合类型。
- 后端：
  - [backend/routes/books.ts](../backend/routes/books.ts) 不改。
  - 不新增 query 参数，不改变 API contract。
- 交互细节：
  - 搜索建议采用去首尾空格、大小写不敏感匹配。
  - 本轮排序只做单字段升序，避免扩成多字段、多方向组合筛选。

### Done when

- 图书页可输入关键字并实时缩小列表。
- 图书页可切换排序字段，排序结果稳定可预期。
- 搜索与排序可同时生效。
- 无匹配时有明确提示，清空搜索词后能恢复完整列表。
- 已收藏标识在搜索或排序后仍正确显示。

## 交付验收标准

- 已登录用户可以在收藏页移除任意一本已收藏书籍，刷新后结果保持一致。
- 已登录用户可以一键清空收藏，刷新后结果保持一致。
- 未登录用户无法绕过现有鉴权直接操作收藏接口。
- 图书页支持按书名和作者排序，默认仍可回到原始顺序。
- 图书页支持按书名和作者实时搜索，输入变化后无需二次提交。
- 搜索与排序不会修改后端图书数据，只影响当前页面展示。
- 现有新增收藏、登录、进入图书页与收藏页流程不回归。

## 风险

- JSON 文件读写是同步覆盖写入，若后续出现并发点击或并发请求，收藏写入存在覆盖风险。本轮只能通过前端禁用重复点击和后端幂等处理降低风险，无法从根上解决文件存储并发问题。
- 当前前端没有现成的单元测试运行基础设施，见 [frontend/package.json](../frontend/package.json)。如果需要补齐前端 L1 和 L2 自动化，可能需要新增依赖，需单独审批。
- favoritesSlice 当前以“操作后重新拉取列表”为主，网络往返会略多，但实现简单、回归面小。本轮优先稳定，不优化为本地乐观更新。
- 搜索和排序如果直接写进全局 store，容易把视图状态与数据状态耦合。本计划已避免这种做法，但实现时需要保持克制。
- 清空全部收藏属于批量破坏性操作，如不做确认，误触风险会上升；如做确认，又会增加交互成本。本轮建议先不扩展复杂确认流，只保留最小保护。

## 边界条件

- 删除单本收藏时，如果目标书籍已不在收藏中，接口应保持幂等，前端不报错。
- 清空全部收藏时，如果当前已为空，接口应保持幂等。
- 用户 token 失效、缺失或对应用户不存在时，接口应沿用现有鉴权和错误码风格。
- 删除最后一本收藏后，页面应进入空状态，而不是停留在旧列表。
- 图书搜索为空字符串时，应展示完整列表。
- 图书搜索无结果时，应展示“无匹配结果”提示。
- 排序切换不应破坏收藏心形标记或按钮文案判断。
- 默认排序应保留后端原始返回顺序，避免用户失去对当前列表顺序的预期。

## 回滚关注点

- 后端新增接口应采用增量方式实现，不修改现有 GET /favorites 和 POST /favorites 契约。这样即使需要回滚，也可以先撤掉前端入口，不影响旧功能。
- 图书搜索和排序只在前端生效，不改动 [backend/routes/books.ts](../backend/routes/books.ts)，回滚成本低。
- favoritesSlice 如新增操作状态字段，应避免影响现有 fetchFavorites 和 addFavorite 的成功路径，否则会带来历史功能回归。
- 回滚时优先顺序建议为：
  - 先隐藏或撤回前端按钮与控件。
  - 再视情况撤回后端 DELETE 接口。
  - 保留原有 GET 和 POST 行为不变。
- 若测试环境数据被批量清空，需要确保沿用现有测试数据拷贝流程，避免污染基线数据，见 [backend/tests/copy-test-data.sh](../backend/tests/copy-test-data.sh)。

## 测试计划初稿

### 测试目标概览

- 以 L0、L3、L4 作为本轮强制交付门槛，补齐“移除收藏、清空收藏、图书排序、图书实时搜索”四项需求的关键闭环。
- 显式覆盖幂等性、刷新持久性、搜索边界和现有新增收藏路径回归。

### L0 静态质量检查

- 后端 TypeScript 类型检查通过，确保新增 DELETE favorites 接口与现有鉴权类型不冲突。
- 前端 TypeScript 类型检查通过，确保新增搜索词、排序项、收藏操作状态不会破坏现有 store 和页面渲染。
- 前后端 ESLint 检查通过。
- 前后端构建通过，确认新增按钮、搜索框、排序控件和异步 action 不引入编译或打包错误。
- 回归关注点：新增删除/清空状态字段后，现有新增收藏和拉取收藏链路仍可正常编译和运行。

### L1 单元测试（建议项 / 条件项）

- 说明：前端当前没有现成单元测试基础设施，且新增依赖需要审批，因此前端 L1 自动化仅作为建议项，不作为本轮强制门槛。
- 条件项：如果本轮把搜索与排序逻辑抽成纯函数或 selector，建议补纯函数测试，不新增依赖也不改变运行时架构。
- 搜索逻辑：书名命中、作者命中、大小写不敏感、输入首尾空白会被裁剪、空字符串返回全量、无匹配返回空结果。
- 排序逻辑：默认顺序保持原始列表顺序、按书名排序、按作者排序、搜索与排序组合后结果仍正确。
- 排序稳定性：对相同作者的数据保持原始相对顺序不变；若后续补纯函数测试，额外使用构造数据覆盖“相同书名”的稳定排序场景。
- 收藏数据变换：若后端把“移除单项收藏”“清空收藏”抽成纯函数，建议覆盖存在项移除、不存在项重复移除、空数组清空、重复清空幂等。

### L2 集成测试（建议项 / 条件项）

- 说明：前端当前无现成组件测试基础设施，本轮不新增依赖时，前端 L2 不作为强制交付项。
- 后端路由集成：继续沿用 Jest + Supertest，覆盖 favorites 路由、鉴权和测试 JSON 文件读写的组合行为。
- 条件项：若后续已有可复用的前端测试基线，可补“组件 + store + API mock”集成测试，覆盖收藏页移除、清空、图书页搜索、排序，以及新增收藏回归。
- 若本轮不落地前端 L2，则必须由 L4 端到端测试补齐上述关键用户路径。

### L3 系统 / API 测试

- GET /favorites 回归：已登录用户可获取收藏列表；未登录返回 401；用户不存在返回 404。
- POST /favorites 回归：已登录用户可新增收藏；重复新增同一本书不产生重复数据；未登录返回 401；用户不存在返回 404。
- DELETE /favorites/:bookId：
  - 已登录且目标已收藏时删除成功，随后再次 GET /favorites 不再返回该书。
  - 已登录但目标不在收藏中时行为幂等，重复调用不报错且数据不被破坏。
  - 未登录返回 401。
  - 用户不存在返回 404。
- DELETE /favorites：
  - 已登录时清空成功。
  - 空列表再次清空行为幂等。
  - 未登录返回 401。
  - 用户不存在返回 404。
- 批量操作后的持久性：单本删除和清空全部后重新读取收藏列表，结果与磁盘持久化一致，不出现刷新回弹。
- 现有新增收藏路径回归：在执行过单本删除或清空全部后，POST /favorites 仍可把图书重新加入收藏，确保新增路径未被新接口破坏。

### L4 端到端测试

- 收藏单本移除：
  - 登录。
  - 从图书页新增一本收藏。
  - 进入收藏页移除该书。
  - 验证列表消失，浏览器硬刷新后仍不存在。
- 收藏单本移除回归：
  - 移除后返回图书页。
  - 确认该书可再次执行“新增收藏”并成功出现在收藏页，防止新增路径被回归破坏。
- 清空全部收藏：
  - 登录。
  - 先保证收藏列表中至少有多本书。
  - 点击清空全部。
  - 验证收藏页进入空状态，硬刷新后仍为空。
- 清空全部收藏后的页面稳定性：
  - 清空完成后，“清空全部收藏”入口应隐藏或禁用。
  - 再次进入收藏页时页面无报错、无脏状态。
- 图书搜索：
  - 登录进入图书页。
  - 按书名输入关键字，列表实时缩小。
  - 按作者输入关键字，结果仅保留匹配书籍。
  - 搜索匹配应大小写不敏感。
  - 输入带首尾空白时结果与裁剪后的关键字一致。
  - 清空输入后恢复完整列表。
  - 输入无匹配关键字时显示无结果提示。
- 图书排序：
  - 登录进入图书页。
  - 选择按书名排序，验证前几项顺序变化符合预期。
  - 切换按作者排序，验证顺序变化符合预期。
  - 使用当前固定数据中“同一作者多本书”的场景验证稳定排序。
  - 搜索与排序组合后结果仍一致且可解释，不出现空白页、重复项或错乱顺序。
  - 排序后收藏标识仍正常显示；在过滤后的结果中新增收藏，进入收藏页后可看到对应书籍。

### 测试数据预置建议

- 后端 API 测试继续使用现有测试数据拷贝流程，确保每轮执行前恢复固定基线，避免删除/清空用例相互污染。
- E2E 继续使用独立注册用户或独立登录用户，避免并发运行时共享收藏状态。
- 清空全部收藏场景优先使用“已有多本收藏”的固定用户；空列表幂等场景优先使用“收藏为空”的固定用户。
- 搜索建议至少使用一组书名关键字、一组作者关键字和一组无匹配关键字，分别验证命中、大小写、空白裁剪和空结果。
- 作者排序稳定性可直接利用现有固定数据中的重复作者。
- 书名排序稳定性若要做自动化验证，建议仅在 L1 条件项中使用构造数据覆盖，因为现有固定图书数据没有重复书名，不适合在 E2E 中硬验证该场景。

## 技术评估约束

- 当前项目基于文件读写持久化，不适合在本轮把收藏操作扩展为高并发、强事务语义设计。
- 本轮不建议新增服务端搜索、排序、分页接口，否则会把简单前端增强演变为 API 契约扩张。
- 本轮不建议引入新状态管理库、查询缓存库或弹窗组件库，收益不足。
- 本轮不建议修改 Book、User 数据结构，避免触发数据迁移和额外兼容逻辑。
